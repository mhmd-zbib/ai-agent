Introduction to Retrieval-Augmented Generation

Retrieval-Augmented Generation (RAG) is an architecture that enhances large language
models by grounding their responses in external knowledge retrieved at inference time.
Instead of relying solely on parametric knowledge baked into model weights during
training, RAG systems query a vector store to fetch the most semantically relevant
passages and inject them as context into the prompt.

This approach addresses two critical limitations of standard LLMs: knowledge cutoff
dates and hallucination. Because retrieved documents are sourced from a live index,
the model can answer questions about events and facts that postdate its training
data. Because the answer is anchored to real passages, the model is less likely to
fabricate plausible-sounding but incorrect information.

---

Core Pipeline Architecture

A RAG ingestion pipeline typically consists of four sequential stages, each
decoupled from the others via a message queue.

Stage 1 — Parsing
Documents arrive as raw bytes (PDF, DOCX, plain text, Markdown). The parser
extracts clean text, preserving structural signals such as headings, paragraph
boundaries, and page numbers. For PDFs, per-page text is retained so that each
downstream chunk can be attributed back to a specific page — a critical feature
for citation and auditability. Noise is removed: headers, footers, repeated
boilerplate, encoding artifacts. The cleaned text is written to object storage
(not passed through the message queue) to keep message payloads small.

Stage 2 — Chunking
Long documents must be divided into segments that fit within the context window
of the embedding model. A sliding-window chunker tokenises the full text using
the same tokeniser as the embedding model (typically cl100k_base for OpenAI
embeddings) and emits windows of a fixed token count — commonly 256 to 512
tokens — with a configurable overlap of 50 to 100 tokens between consecutive
windows. The overlap ensures that information spanning a window boundary is
represented in at least one chunk. Each chunk is assigned a stable, deterministic
identifier derived from the document ID and its position. The total chunk count
is recorded in Postgres so the pipeline can determine when all chunks have been
processed.

Stage 3 — Embedding
Each chunk travels as an independent message to the embedding stage. This
decoupling is intentional: the embedding model may be rate-limited, and fanning
out one message per chunk allows multiple embedding workers to process chunks in
parallel. The embedding service calls the configured model endpoint (OpenAI,
Ollama, Azure OpenAI, or any OpenAI-compatible provider) and receives back a
dense float vector. The provider is injected as a protocol dependency so the
embedding stage contains no provider-specific code. The vector and the original
chunk text are forwarded together to the store stage.

Stage 4 — Storage
The store stage upserts each vector into the vector database alongside rich
metadata: document ID, chunk ID, chunk index, source page, file type, tenant
identifier, and the chunk text itself. Storing the chunk text inline in the
vector record makes retrieval self-contained — a similarity search returns
everything needed to construct an answer without an extra Postgres round-trip.
Once the number of stored chunks equals the total chunk count recorded in Stage
2, the document is marked completed.

---

Vector Database Design

The choice of vector database determines the operational complexity of the system.

Self-hosted options such as Qdrant, Weaviate, and Milvus provide full control
over data residency, query latency, and cost at the expense of operational burden.
Qdrant is particularly well-suited for small-to-medium deployments because it ships
as a single binary with no external dependencies, supports gRPC and REST, and
offers first-class payload filtering that enables multi-tenant isolation in a
single collection.

Cloud-hosted options such as Pinecone abstract away all infrastructure concerns
but introduce vendor lock-in and per-query pricing that can become significant at
scale. Pinecone's namespace feature provides native tenant isolation, mapping
cleanly to a user_id namespace per tenant.

A production system should treat the vector backend as a pluggable adapter
behind a protocol interface so that the choice of provider can be changed via
configuration without modifying application logic.

---

Tenant Isolation

Multi-tenant RAG systems must ensure that one user's documents cannot appear in
another user's query results. Two main strategies exist.

Namespace-per-tenant: the vector database natively partitions the index by
namespace. Pinecone and Weaviate support this directly. Queries are scoped to
a single namespace, so cross-tenant leakage is impossible at the database level.

Payload filtering: all vectors share a single collection or index, and each
vector carries a user_id field in its metadata. Every query includes a mandatory
filter on this field. This is the pattern used with Qdrant when running in
single-collection mode. It is slightly less isolated than namespace-per-tenant
because the filter must be applied correctly at every query site, but it is
simpler to operate because collection management is centralised.

---

Failure Handling and Observability

Each pipeline stage should update the document status in Postgres at both the
start and end of its work. This provides a real-time audit trail and enables
dead-letter queue (DLQ) handlers to correlate a failed message with the document
that caused it. Status values form a linear progression:

  uploaded → parsing → parsed → chunking → chunked →
  embedding → storing → completed

If any stage fails, the document transitions to failed and the error message is
recorded. The message is nacked to the stage's DLQ rather than requeued, so
transient infrastructure failures do not cause infinite retry loops. A separate
DLQ consumer or manual re-queue operation can replay failed messages after the
root cause is resolved.

Observability tooling should expose at minimum: message queue depths per stage,
DLQ depths, per-document status and error, and embedding API latency and error
rates. These four signals are sufficient to diagnose the vast majority of
operational issues in a RAG ingestion pipeline.
